1 Executive summary
BuildProClinic (MedFlow)— a three-tier, Django + DRF-based Hospital Management
System (HMS) that digitizes clinical workflows (EHR, appointments, prescriptions, billing),
enforces role-based access control (RBAC), logs auditable changes, and provides a Research
& Blog portal for doctors to publish internal research. The system will be extensible, secure,
testable, and demonstrable for semester submission.

5 Functional requirements
Each feature includes a brief description and acceptance criteria.
5.1 Authentication & RBAC
Description:Register/login, password management, roles and server-side permissions.
Acceptance:
•Admin can create users and assign roles.
•Doctor can view/edit own patients (unless admin override).
•Unauthorized requests return 403 or redirect to login.
5.2 Patient EHR
Description:CRUD patient profiles, medical history, allergies, visit records, lab report at-
tachments.
Acceptance:
•Create patient record with core demographics.
•Visit entries persist with timestamp and author.
•Attach/preview lab report PDFs (DEV: local MEDIA).
5.3 Appointment Management
Description:Book/reschedule/cancel appointments; show doctor availability; conflict preven-
tion.
Acceptance:
•Patients or receptionists can create appointments within doctor’s slots.
•System prevents double-booking and shows conflict warning.
•Doctors see daily calendar view.
•Note:System logs a “notification sent” event to the console/audit log when booked (Real
Email/SMS integration is out of scope).
5.4 Prescription Management
Description:Create digital prescriptions (medicines, dosage, instructions) and export PDF.
Acceptance:
•Create prescription linked to patient and visit.
•Export printable PDF.
•Pharmacy/authorized roles can view read-only.
5.5 Billing & Invoicing
Description:Generate invoices for consultations and tests; mark payment status.
Acceptance:
•Create invoice line items, compute totals, mark paid/unpaid.
•Admin or billing role can view revenue summary.
4
5.6 Audit Logging
Description:Record who/what/when for sensitive CRUD operations.
Acceptance:
•CRUD on patient/prescription/billing recorded with user id, timestamp, changes (if pos-
sible).
•Admin can query recent audit logs.
5.7 Research & Blog Portal
Description:Doctors draft/publish blog posts and upload paper PDFs; admin approval re-
quired.
Acceptance:
•Doctor uploads paper (PDF) with metadata (title, authors, abstract).
•Upload queued for admin approval; approved items visible in public list.
•Admin can reject/approve and add notes.
5.8 API Layer
Description:DRF endpoints (CRUD + filters + pagination) for core models.
Architecture Note:The Web UI will primarily use Django Templates with direct ORM
access for development speed. The REST API is implementedin parallelto support future
mobile/SPA clients.
Acceptance:
•Endpoints require auth where appropriate; return JSON and support pagination.
•Support query parameters for filtering: e.g.,GET /api/patients/?search=nameorGET
/api/appointments/?date=2026-01-20.
6 Non-functional requirements
•Security:RBAC enforced server-side; secure password hashing; audit logs append-only;
file uploads validated for MIME and size.
•Privacy:Patient data treated as sensitive; avoid exposing PII in public endpoints or
logs.
•Performance:Responsive for up to∼50 concurrent dev users (no heavy tuning).
•Maintainability:ModularDjangoapps(accounts,ehr,appointments,billing,publications,
audit).
•Reliability:Repeatable migrations; use DB backups in production.
•Usability:Simple role-specific UIs; client- and server-side validation.
•Reproducibility:requirements.txtandREADMEwithsetupsteps(WSLinstructions
included).

12 Security & compliance
•Validate and limit uploads (max file size, allowed MIME types).
•Use HTTPS in production; secure cookies (SESSION_COOKIE_SECURE).
•Role-based permission checks on UI and API.
•Audit logs append-only (immutable).
•Sanitize all user input; escape or sanitize rich text for blogs.
13 Risks & mitigations
•Dependency incompatibility— pin versions inrequirements.txt, use virtualenv.
•File-upload malware— admin approval + mimetype checks for uploaded papers (dev);
consider virus scanning for production.
•Time constraints— implement core features first; treat research/blog as optional if
time-limited.
•DB migration issues— keep migrations small and tested; backup DB before major
changes.
